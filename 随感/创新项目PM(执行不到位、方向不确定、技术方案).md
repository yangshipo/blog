# 三个核心问题的本质区别

## 一句话总结

| 问题 | 本质 | 关键特征 | PM的应对 |
|------|------|----------|----------|
| **执行不到位** | 人的问题 | 知道要做什么，但做不好/做偏了 | 建立反馈机制 |
| **方向不确定** | 目标的问题 | 不知道该做什么才是对的 | 快速验证假设 |
| **技术方案** | 手段的问题 | 知道做什么，但不知道怎么做最好 | 分层决策 |

---

## 详细对比

### 1. 执行不到位

**典型表现：**
```
你说：做一个登录页面，要支持手机号登录
结果：做出来只支持邮箱登录

你说：这个按钮要放在右上角
结果：放在了左下角

你说：这周五要上线
结果：下周一还没做完
```

**根本原因：**
- ❌ 不是能力不够（会做，但没做对）
- ❌ 不是不努力（在做，但理解偏了）
- ✅ **信息传递失真**（你说的和他理解的不一样）

**如何判断是这个问题：**
- 事后复盘时，团队说"我以为是XXX"
- 做出来的东西和你想的完全不一样
- 同样的错误反复出现

**PM的应对：**
```
核心：缩短反馈周期

传统：需求 → 2周后看结果 → 发现做错了
优化：需求 → 2天看Demo → 发现理解偏差 → 立即纠正

具体做法：
- 每天站会（昨天做了什么，今天做什么，有什么问题）
- 每周Demo（强制产出可演示的东西）
- 用真实场景验证（不要等到上线才发现问题）
```

---

### 2. 方向不确定

**典型表现：**
```
做了一个功能，用户不买账
做了A方案，老板说要改成B方案
做了2个月，发现整个方向错了要推倒重来
```

**根本原因：**
- ❌ 不是执行问题（做得很好，但方向错了）
- ❌ 不是技术问题（技术很牛，但没用对地方）
- ✅ **假设未验证**（以为用户要A，其实用户要B）

**如何判断是这个问题：**
- 做出来的东西没人用
- 数据指标没有达到预期
- 团队经常争论"到底该做什么"

**PM的应对：**
```
核心：用最小成本验证假设

错误做法：花3个月做一个完美的产品 → 上线后发现方向错了
正确做法：花3天做一个Demo → 给10个用户试用 → 根据反馈调整

具体做法：
- 明确当前要验证的假设（"我们假设用户需要XXX"）
- 设计最小实验（用最简单的方式验证）
- 设定验证标准（什么数据证明假设成立/不成立）
- 快速决策（1周内决定继续/调整/放弃）
```

**示例：**
```
假设：用户需要AI生成代码功能

最小实验：
- 不做产品，先用GPT-4手动给3个用户生成代码
- 观察他们是否愿意用、用得怎么样
- 如果3个人都说好，再投入资源做产品
- 如果3个人都不用，立即调整方向

而不是：
- 花2个月做一个完整的AI代码生成平台
- 上线后发现没人用
- 浪费了2个月
```

### 3. 技术方案

**典型表现：**
```
用React还是Vue？
用微服务还是单体架构？
用MySQL还是MongoDB？
要不要引入Kubernetes？
```

**根本原因：**
- ❌ 不是方向问题（知道要做什么）
- ❌ 不是执行问题（知道怎么做）
- ✅ **选择太多**（有多种实现方式，不知道选哪个）

**如何判断是这个问题：**
- 团队在技术选型上争论不休
- 担心"选错了技术栈，以后很难改"
- 纠结于"要不要用最新的技术"

**PM的应对：**
```
核心：分层决策，不要一次性决定所有事情

错误做法：
- 一开始就确定完整的技术架构
- 担心以后扩展，现在就做复杂设计
- 追求技术先进性

正确做法：
- 只决定当前阶段必须决定的事情
- 其他的等需要时再决定
- 优先选择团队熟悉的技术

具体做法：
Layer 1（必须现在决定，6个月不变）
- 编程语言（Java/Python/Go）
- 核心框架（React/Vue）
- 数据库类型（SQL/NoSQL）

Layer 2（1-2个月后再决定）
- 具体的库和工具
- API设计
- 部署方式

Layer 3（随时可以改）
- 代码结构
- 命名规范
- 具体实现细节
```

**示例：**
```
营销组件AI出码项目：

现在必须决定：
✅ 用GPT-4（因为代码生成质量最好）
✅ 支持React（因为公司90%项目用React）
✅ 用Figma API（因为设计师都用Figma）

现在不用决定：
❌ 要不要支持Vue（等React验证成功再说）
❌ 要不要做在线编辑器（先下载代码，需要时再做）
❌ 要不要支持单元测试生成（先保证基础功能，再扩展）
```

## 三者的关系

```
方向不确定
    ↓
技术方案摇摆（不知道该为哪个方向优化）
    ↓
执行更混乱（团队不知道该做什么）
```

**恶性循环：**
1. 方向不清楚 → 技术方案定不下来
2. 技术方案变来变去 → 团队执行混乱
3. 执行混乱 → 做出来的东西不对 → 方向更不确定

**破局点：**
```
先解决方向问题（用最小实验验证）
    ↓
方向清楚了，技术方案自然就清楚了
    ↓
技术方案清楚了，执行就不会乱
```

---

## 实战判断方法

**当团队出现问题时，问自己三个问题：**

### 问题1：我们知道要做什么吗？

- **YES** → 不是方向问题
- **NO** → 是方向问题，需要快速验证假设

### 问题2：我们知道怎么做吗？

- **YES** → 不是技术方案问题  
- **NO** → 是技术方案问题，需要分层决策

### 问题3：做出来的东西符合预期吗？

- **YES** → 没问题，继续
- **NO** → 是执行问题，需要缩短反馈周期

---

## 给PM的快速决策表

| 症状 | 问题类型 | 立即行动 |
|------|----------|----------|
| 做出来的东西和想的不一样 | 执行不到位 | 明天开始每日站会 |
| 做出来了但没人用 | 方向不确定 | 这周找3个用户深度访谈 |
| 团队争论用什么技术 | 技术方案 | 只决定Layer 1，其他延后 |
| 需求经常变 | 方向不确定 | 定一个北极星指标，所有需求用它判断 |
| 进度总是延期 | 执行不到位 | 把任务拆小，2天一个检查点 |
| 不知道该做什么功能 | 方向不确定 | 做一个最小Demo，给用户试用 |

---

**记住：**
- **执行不到位** → 建立反馈机制（让问题早暴露）
- **方向不确定** → 快速验证假设（用数据说话）
- **技术方案** → 分层决策（不要过早优化）

**三个问题要按顺序解决：先方向，再技术方案，最后执行。**
